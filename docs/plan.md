# FamilyVault ‚Äî Full Migration Plan

## Context

~2TB of photos/videos exist on both iCloud and Google Photos. The goal is to download
everything to a Mac Mini home server with 12TB RAID1 storage, preserve all metadata,
and run Immich as a self-hosted photo app. Ongoing sync is required.

Both accounts are merged into an **iCloud Shared Photo Library**. Wife is the primary
photo producer (~80% of photos are hers).

---

## Key Decision: iCloud as Primary Source

**iCloud wins over Google for the primary download.**

| Factor | iCloud | Google Photos |
|--------|--------|---------------|
| Original quality | Always exact originals | Only if "Original Quality" was set |
| HEIC preservation | Yes | Depends on upload method |
| Live Photos | Native paired HEIC+MOV | Split into separate files |
| Metadata (EXIF) | Embedded in files | Stripped into sidecar JSON (broken) |
| Edited photos | Original + edit preserved | Usually only edited version |
| Export tooling | osxphotos (excellent) | Takeout + post-processing (painful) |
| Ongoing sync | Yes (osxphotos --update) | No (API dead since March 2025) |

**Strategy: Full iCloud export as primary ‚Üí wife's metadata injection ‚Üí Google Takeout only for delta.**

## Metadata Coverage in the Shared Library

Two assumptions were tested and disproven during IMP-011 and the spec-013 research.

### Assumption 1 (DISPROVEN): "iCloud strips GPS from shared copies at scale"

Wrong. Most no-GPS photos never had GPS in the first place ‚Äî Location Services was
off for Alice's iPhone Camera, plus any photo saved from Telegram / WhatsApp /
Instagram / screenshots arrives without GPS by design. iCloud only stripped GPS in a
narrow set of cross-library cases ‚Äî about **37 UUIDs / 24 files** out of ~80k.

Why no GPS ‚Äî by source (Alice's library):

| Source | Count | Reason |
|--------|-------|--------|
| Camera app photos *with* GPS | ~1,400 | Location Services on |
| Camera app photos *without* GPS | ~12,400 | **Location Services for Camera disabled** on her iPhone |
| Photos saved from messaging apps (Telegram, WhatsApp, Instagram, Messenger) | ~7,000 | Messaging apps strip GPS before sending |
| Screenshots | ~1,400 | Never have GPS |
| Photos shared between you and wife with strip | 37 UUIDs / 24 files | iCloud cross-library strip ‚Äî narrow |

### Assumption 2 (DISPROVEN): "Wife's metadata is not shared with me automatically"

Wrong. **iCloud Shared Photo Library propagates ALL metadata to all participants.**
Verified empirically (spec 013 research, 2026-04-25):

| Metric | My library | Alice's library | Œî |
|--------|-----------|-----------------|---|
| Shared assets | 79,853 | 79,882 | 29 (sync lag) |
| **Shared with GPS** | **6,663** | **6,663** | **0** |
| **Favorited (shared)** | **1,885** | **1,885** | **0** |
| **Description set (shared)** | 223 | 223 | 0 |

Cross-library sample on 30 GPS photos by date-bridge: **30/30 GPS values
byte-for-byte identical**. Exhaustive set-difference check on all 6,663 GPS dates:
**0 photos in either direction had GPS the other didn't**.

Library ‚Üí export file is also in sync: 100/100 random shared+GPS photos have GPS
in their export file's EXIF. `osxphotos --update --exiftool` (the daily sync at
2 AM via launchd) correctly catches metadata-only updates and re-writes them
into export files.

### What this means

iCloud Shared Library + daily `osxphotos --update` covers the metadata pipeline
end-to-end. No manual import script is required for the steady state.

### Defensive scripts (kept for cold-start and edge cases)

- **`apply-wife-metadata.py`** ‚Äî date-based bridge from wife's `Photos.sqlite` to
  the export. Originally written under IMP-011 for a hypothesized GPS gap; updated
  24 files in production. Now understood to have caught a transient sync race; the
  iCloud sync resolved the same gap independently within ~2 days. Kept as a
  defensive cold-start tool; no longer required for ongoing operation.

- **`infer-gps.py`** ‚Äî for shared-copy "(N)" duplicates that for some reason still
  lack GPS, infer it from temporal-neighbor photos in your own library. Same
  status: rarely needed, retained as a safety net.

- **People/face tags**: never imported. Photos.app face data is locally generated by
  `photoanalysisd`; only 22 photos in Alice's library had named persons. Immich runs
  its own face ML and produces better results.

> Detailed evidence and method for the second finding is in
> `specs/013-wife-metadata-import/research.md`.

## Phase 1: iCloud Full Export

**Status**: ‚úÖ Complete ‚Äî 381,310 files exported (IMP-011 re-export run)

### Run Export

```bash
# Runs in tmux ‚Äî survives SSH disconnect
ssh macmini
export PATH=/opt/homebrew/bin:$PATH
tmux new -s icloud-export
cd ~/projects/takeout/takeout
./scripts/export-icloud.sh \
  /Volumes/HomeRAID/icloud-export \
  "/Volumes/HomeRAID/Photos Library.photoslibrary"
```

The `--update` flag resumes from where it left off. `--update-errors` retries previously
failed files. Re-running this script is always safe.

**Monitor progress:**
```bash
tail -f /Volumes/HomeRAID/icloud-export/osxphotos-export-*.log
```

**Expected duration**: 3‚Äì5 hours unattended (first run); minutes on incremental re-runs.

**Before starting**: stop Immich completely to free CPU:
```bash
cd setup/immich && docker compose down
```

### What export-icloud.sh does (IMP-011 final form)

- `--library` explicit path ‚Üí no ambiguous library detection
- `--update --update-errors` ‚Üí incremental + retry previously failed files
- `--exiftool --exiftool-option '-m'` ‚Üí writes Photos.app metadata (GPS where available)
- `--fix-orientation` ‚Üí corrects EXIF rotation; sidesteps Rotate-180 bugs
- `--export-edited` ‚Üí exports processed HEIC alongside DNG (fixes "dark ProRAW" issue)
- `--sidecar xmp --sidecar json --person-keyword --album-keyword` ‚Üí full metadata sidecars
- `--touch-file` ‚Üí sets file mtime to capture date; also makes Immich re-index changed files

**Note**: GPS coverage in your shared library is mostly *upstream*-determined ‚Äî see
"Metadata Coverage" above. Most no-GPS photos never had GPS to begin with (Location
Services disabled on Alice's iPhone Camera, photos saved from messaging apps, etc.).

## Phase 2: Wife's Metadata Extraction and Injection (defensive / optional)

**Status**: ‚úÖ Complete ‚Äî 24 files updated in IMP-011 production run.

> **Important**: research in spec 013 (see `specs/013-wife-metadata-import/research.md`)
> showed that the original premise ‚Äî *Alice's metadata isn't shared with my library* ‚Äî
> is wrong. iCloud Shared Library propagates GPS, favorites, descriptions, etc. between
> participants automatically. The 24 files this script updated represent a transient
> sync race that iCloud would have resolved independently within ~2 days.
>
> This phase is **no longer required for ongoing operation**. Kept here as a
> cold-start defensive tool: if a future sync race or new full export reveals a
> gap, this is the patch path.

### Why

A small subset of shared-library copies briefly have GPS in wife's library but not
yours during cross-device sync. `apply-wife-metadata.py` reads her `Photos.sqlite`
and patches matching files in the export. In steady state (after iCloud sync
catches up), this script is a no-op.

**Why date-based, not UUID-based**: iCloud Shared Library reassigns UUIDs across libraries
‚Äî the wife's library and your shared library use *different* UUIDs for the same shared
photo. Direct UUID match fails. The script bridges via `ZDATECREATED` (capture time):

```
wife's DB (date ‚Üí GPS)
    ‚Üí shared library DB (date ‚Üí UUID)
    ‚Üí export DB (UUID ‚Üí filepath)
```

### Step 1 ‚Äî Get wife's Photos.sqlite (15 min)

On Mac Mini, create a second macOS user for wife's Apple ID:
1. System Settings ‚Üí Users & Groups ‚Üí Add User
2. Log in as her
3. **Before opening Photos.app**: System Settings ‚Üí Apple ID ‚Üí iCloud ‚Üí Photos ‚Üí
   confirm **"Optimize Mac Storage"** is selected (prevents downloading 2TB of originals)
4. Open Photos.app ‚Üí wait until photo count stabilizes (5‚Äì15 min ‚Äî only metadata syncs)
5. Make her library readable from your account (one-time):
   ```bash
   sudo cp -R "/Users/alice/Pictures/Photos Library.photoslibrary" /Volumes/HomeRAID/alice/
   sudo chown -R szelenin:staff /Volumes/HomeRAID/alice/
   ```
6. Log out of her account

### Step 2 ‚Äî Run metadata injection

```bash
# Dry run first ‚Äî shows what would be updated
python3 scripts/apply-wife-metadata.py \
  "/Volumes/HomeRAID/alice/Photos Library.photoslibrary/database/Photos.sqlite" \
  /Volumes/HomeRAID/icloud-export \
  --shared-db "/Volumes/HomeRAID/Photos Library.photoslibrary/database/Photos.sqlite" \
  --dry-run

# Apply
python3 scripts/apply-wife-metadata.py \
  "/Volumes/HomeRAID/alice/Photos Library.photoslibrary/database/Photos.sqlite" \
  /Volumes/HomeRAID/icloud-export \
  --shared-db "/Volumes/HomeRAID/Photos Library.photoslibrary/database/Photos.sqlite"
```

**What it injects** (date-based bridge, photos only ‚Äî videos and sidecars skipped):

| Field | Source (wife DB) | EXIF tag written |
|-------|------------------|-----------------|
| GPS coordinates | `ZASSET.ZLATITUDE/ZLONGITUDE` | `GPSLatitude/Longitude/Refs` |
| Title | `ZADDITIONALASSETATTRIBUTES.ZTITLE` | `Title`, `XMP:Title` |
| Description | `ZADDITIONALASSETATTRIBUTES.ZASSETDESCRIPTION` | `Description`, `Caption-Abstract` |
| Timezone | `ZADDITIONALASSETATTRIBUTES.ZTIMEZONENAME` | `OffsetTimeOriginal` |
| Keywords | `ZKEYWORD` join | `Keywords+=` (additive) |

**Production result**: 37 UUID matches ‚Üí 24 photo files updated. Most "no GPS" photos
never had GPS in either library (Location Services disabled, social-media saves, etc.) ‚Äî
not stripped by iCloud as originally hypothesized.

### Step 3 ‚Äî Backfill any remaining gaps with infer-gps.py

For shared-copy "(1)" / "(2)" duplicates that still have no GPS, infer from temporal
neighbors in your own library:

```bash
python3 scripts/infer-gps.py /Volumes/HomeRAID/icloud-export --dry-run
python3 scripts/infer-gps.py /Volumes/HomeRAID/icloud-export
```

Default temporal gap is 1 hour; pass `--max-gap 10800` for 3-hour windows on travel days.

## Phase 3: Immich Setup

**Status**: ‚úÖ Complete ‚Äî 93,007 photos + 53,303 videos = 146,310 assets indexed

### Start Immich

```bash
cd setup/immich
docker compose up -d
```

Open `http://macmini:2283` ‚Üí complete first-run setup (create admin user, save API key
to `/Volumes/HomeRAID/immich/api-key.txt`).

### Add external library ‚Äî IMPORTANT: set DNG exclusion BEFORE first scan

The export contains both ProRAW DNG and the processed HEIC sibling. Indexing both
duplicates every ProRAW shot and shows the dark/raw DNG as the gallery preview.
Exclude DNG (and other raw formats) before any scan runs:

```bash
IMMICH_KEY=$(cat /Volumes/HomeRAID/immich/api-key.txt)

curl -X POST http://localhost:2283/api/libraries \
  -H "x-api-key: $IMMICH_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "iCloud Export",
    "importPaths": ["/usr/src/app/icloud-export"],
    "exclusionPatterns": ["**/*.DNG","**/*.dng","**/*.RAW","**/*.raw","**/*.CR2","**/*.cr2","**/*.ARW","**/*.arw"]
  }' | jq '{id, name}'
```

### Trigger scan

```bash
LIBRARY_ID="<id from above>"
curl -X POST "http://localhost:2283/api/libraries/$LIBRARY_ID/scan" \
  -H "x-api-key: $IMMICH_KEY"
```

Immich indexes all files; ML jobs (face detection, CLIP search) run automatically.
For 380k+ files (~150k assets after DNG exclusion) this takes several hours.

## Phase 5: Ongoing Sync

**Status**: ‚úÖ Complete ‚Äî daily at 2 AM via launchd

macOS Sequoia blocks `crontab` modifications without Full Disk Access. Use a launchd
agent instead. The plist + script are already on the RAID:

```
/Volumes/HomeRAID/scripts/sync.sh
~/Library/LaunchAgents/com.familyvault.sync.plist
```

Reload after edits:
```bash
launchctl unload ~/Library/LaunchAgents/com.familyvault.sync.plist
launchctl load   ~/Library/LaunchAgents/com.familyvault.sync.plist
launchctl list | grep familyvault.sync
```

Logs: `/Volumes/HomeRAID/sync.log`.

New photos ‚Üí iCloud ‚Üí Photos.app ‚Üí picked up on next 2 AM run.
Lag: a few hours from shooting to appearing on RAID.

**Note**: wife's metadata injection does NOT need to re-run for new photos ‚Äî current-day
shots come through with GPS intact (when Location Services is on for Camera).

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Export interrupted again | `--update` resumes exactly where it stopped |
| Wife's UUID doesn't match shared copy UUID | Verify with sample query before running injection |
| GPS coverage still low after injection | Check dry-run output; fallback to infer-gps.py for remaining gaps |
| Google Takeout HEIC/JPEG duplicates | Immich built-in duplicate detection cleans up after import |
| RAID failure during migration | Do not delete cloud copies until Phase 6 verification passes |
| Mac wakes from sleep mid-transfer | System Settings ‚Üí Battery ‚Üí Prevent sleep while on power |
